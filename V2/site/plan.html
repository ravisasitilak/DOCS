Guest OS Level Volume Encryption

Guest-level dm-crypt/LUKS encryption with OpenBao as key management backend

What we're doing

Encrypt Cinder volumes inside the guest VM using Linux's native dm-crypt/LUKS. Not at the hypervisor, not at the storage layer. A shell script (luks-agent.sh) runs inside the VM, detects attached volumes, decides whether to encrypt them, and manages keys through OpenBao (Transit for encryption, KV for storage).

Two independent decisions:

  1. How does the script get into the VM? — Golden image or vendor data
  2. Which volumes get encrypted? — All of them, or only specific ones (per-volume via Cinder properties)

These are orthogonal. Any combination works.

Three services — each does one thing

The VM interacts with three separate services. Don't confuse them.

ServiceResponsibilityWhat it does NOT do
IMDS Token minting only. Signs JWT (RSA-4096), calls OpenBao /auth/jwt/login, returns BAO token to the VM. Does NOT query Cinder. Does NOT know about volume types or encryption flags. Does NOT deliver scripts.
Metadata Service Serves volume properties to the VM. At boot: delivers volumes.json via Nova vendor data. On hot-attach: responds to GET /v1/volumes/{uuid} with encryption flag. Authenticates to Keystone, then queries Cinder for volume type extra_specs. Does NOT mint tokens. Does NOT talk to OpenBao.
Keystone Identity service. Issues scoped tokens to the metadata service's service account so it can query Cinder. Standard OpenStack auth — no custom integration. Does NOT know about encryption. Only the metadata service uses it (server-side). The VM never talks to Keystone directly.
OpenBao Key management. Transit engine encrypts/decrypts passphrases. KV store holds ciphertext by volume UUID. Does NOT know about Cinder. Does NOT know which volumes are encrypted.
Clean separation: IMDS handles authentication (BAO tokens). Metadata service handles volume properties (encrypt yes/no) — authenticates to Cinder via Keystone. OpenBao handles key operations (encrypt, decrypt, store). The VM ties them together.

Custom Metadata Server — one mode, same flow for boot and hot-attach

The image already has cloud-init with the OpenStack datasource built in. When a VM boots, cloud-init automatically queries Nova's metadata service at 169.254.169.254 and fetches vendor data (operator-provided: our fetcher script + agent.conf) and user data (tenant-provided: customer's own scripts). cloud-init combines both and runs them. Our vendor data contains a fetcher that downloads and installs luks-agent.

Once luks-agent is running, it queries the custom metadata server per-volume with the volume UUID to check if it needs encryption. Same flow for boot and hot-attach — the agent sends UUIDs, gets back encrypt yes/no. Nova does not pre-fetch or cache volume flags. There is no volumes.json.

The only endpoint: GET /v1/volumes/{uuid}

Verifies VM by source IP → authenticates to Keystone → queries Cinder for the volume's extra_specs → returns {encrypt: true/false}.

Keystone authentication — Custom Metadata Server → Cinder

The custom metadata server is a normal OpenStack service. It authenticates to Keystone like any other service (Nova, Neutron, etc.) to get a scoped token, then uses that token to call Cinder APIs. Standard OpenStack service-to-service auth — nothing custom.

What it needs from Cinder

One read-only API call per volume lookup:

  • GET /v3/types/{type_id}/extra_specs — read volume type extra_specs to check if openbao_encrypt=true

Read-only. The custom metadata server never creates, modifies, or deletes anything in Cinder.

Recommended Option A: Golden image

Everything pre-installed in a Glance image. Build once, use everywhere.

How it works

  1. Boot temp VM from base Ubuntu/CentOS image
  2. Install dependencies: cryptsetup, jq, curl
  3. Copy luks-agent.sh, systemd units, udev rules, agent.conf
  4. Enable luks-agent-boot.service and luks-token-renew.timer
  5. Clean up, snapshot, upload to Glance as ubuntu-encrypted

Pros & cons

  • Fast boot — no package installation at runtime
  • Works without cloud-init
  • Must rebuild image when agent is updated

Option B: Vendor data

Metadata service delivers a lightweight fetcher script to any VM at boot via Nova's vendor data mechanism + cloud-init. The fetcher downloads the actual luks-agent code and systemd units. Vendor data itself stays small (size limits).

How it works

  1. Metadata service is registered as a Nova dynamic vendor data provider
  2. At VM creation, Nova calls the metadata service → it returns vendor_data2.json containing a fetcher script, volumes.json, and agent.conf
  3. Nova serves this via 169.254.169.254
  4. cloud-init inside the VM fetches and processes it automatically
  5. Fetcher script downloads luks-agent.sh, systemd units, and udev rules
  6. Installs packages, writes files, enables services, reloads udev

Pros & cons

  • Works with any Linux image that has cloud-init
  • Agent updates deploy immediately — no image rebuild
  • Vendor data stays small (only fetcher + config + volume flags)
  • First boot is slower (~30-90s extra for package install + download)
Vendor data contains NO secrets and NO actual scripts. Only a fetcher script and agent.conf. The fetcher downloads the real scripts at first boot. The BAO token comes from IMDS at runtime. Volume encryption flags come from the custom metadata server at runtime. No volumes.json.

How the agent knows which volumes to encrypt

There is no volumes.json. The luks-agent queries the custom metadata server per-volume at runtime — same flow for boot and hot-attach.

        VM->>MetaSrv: GET /v1/volumes/{uuid}

Why no volumes.json?

  • No stale data — agent always gets the current state from Cinder via the metadata server
  • Same flow for boot and hot-attach — no special cases
  • Vendor data stays minimal (fetcher + config only)
  • Works with golden image too — agent queries metadata server regardless of how it was installed

Note the service split in this flow: The agent calls IMDS only for BAO tokens. It calls the metadata service for volume properties. It calls OpenBao (with the BAO token) for key operations. Three separate services.

Mode 1: Encrypt all volumes

If encrypt_all = true in agent.conf, the agent skips all property checks. Every attached volume (except root vda) gets LUKS. No metadata service queries needed for volume flags.

# /etc/luks-agent/agent.conf
[luks]
encrypt_all = true

Can't mix encrypted and unencrypted volumes on the same VM. If you need that, use per-volume mode.

Mode 2: Per-volume via Cinder property

If encrypt_all = false, the agent checks each volume before encrypting. The flag comes from Cinder volume type's openbao_encrypt=true property, surfaced via the metadata service.

# /etc/luks-agent/agent.conf
[luks]
encrypt_all = false

Cinder volume type setup (admin, one-time)

openstack volume type create secure-volume
openstack volume type set secure-volume --property openbao_encrypt=true

Customer usage

# This volume gets encrypted
openstack volume create --type secure-volume --size 100 my-secret-data

# This volume does NOT get encrypted
openstack volume create --size 100 my-logs

Plain Cinder type — no --encryption-provider, no Barbican. It's purely a flag.

How the VM authenticates to OpenBao

The VM needs a BAO token to call Transit and KV APIs. This comes from IMDS — which only does token minting:

  1. Nova creates VM → IMDS provides initial BAO token via vendor data
  2. IMDS signs a 60-second JWT (RSA-4096) with claims: project_id, instance_id
  3. IMDS calls OpenBao POST /auth/jwt/login with the JWT
  4. OpenBao validates signature → returns project-scoped token (1h TTL)
  5. Token written to /run/bao-token (tmpfs, 0600)

Token renewal

systemd timer fires every 45 minutes. Calls IMDS POST /v1/renew. IMDS verifies VM by source IP, signs a new JWT, gets a fresh BAO token. 1h TTL with 45-min renewal = 15-minute buffer.

Three keys — don't confuse them

KeyWhat it doesWhere it livesBound to
LUKS master key Encrypts actual data on disk (AES-XTS) LUKS header on the volume (encrypted by passphrase) Volume

The passphrase is stored by volume UUID, not VM identity. Any authorized VM can retrieve it.

Transit vs KV — why both?

Transit: Encryption-as-a-service. Stateless. Stores nothing. The Transit key lives in OpenBao memory only.

KV: Simple key-value database. No encryption logic. Just holds the Transit-encrypted ciphertext.

Why not just store plaintext in KV? Transit protects against: BAO storage dump/backup stolen, over-privileged BAO admins with KV read access, KV policy misconfigurations. If you don't need this extra layer, skip Transit — store plaintext passphrase directly in KV.

How the VM gets the Cinder volume UUID

$ lsblk -ndo SERIAL /dev/vdb
a1b2c3d4-e5f6-7890-abcd-ef1234567890

When Cinder attaches a volume via libvirt/QEMU, the Cinder volume UUID is set as the disk's serial number. Standard OpenStack behavior.

  • Stable across reboots
  • Stable across detach/attach — same UUID on different VMs
  • Stable across device name changes — /dev/vdb might become /dev/vdc

Never use /dev/vdX as an identifier — it changes.

Volume migration between VMs

Works because the passphrase is stored by volume UUID in OpenBao, not by VM identity.

  1. VM-A: umount + cryptsetup close
  2. Detach from VM-A, attach to VM-B
  3. VM-B detects new device, gets same Cinder UUID via lsblk SERIAL
  4. VM-B finds LUKS header → Path B
  5. VM-B reads ciphertext from OpenBao KV — same UUID, same KV path
  6. VM-B calls Transit decrypt → gets original passphrase
  7. VM-B: cryptsetup open + mount

BAO policy must allow VM-B's project to access the volume's KV path.

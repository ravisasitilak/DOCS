Key findings

  1. Guest dm-crypt is 4.4× faster than QEMU LUKS on large-block I/O (1,391 vs 316 MB/s)
  2. QEMU front-end = QEMU back-end — control_location doesn't change the crypto engine in 2024.2
  3. The ~310 MB/s ceiling = one host core saturated at ~125% CPU; dm-crypt uses ~480% (4 cores)
  4. 4k I/O: all methods are within ±10% — encryption is essentially free at small block sizes
  5. dm-crypt p99 latency is close to plain (39 ms vs 34 ms at 1m d=32); QEMU LUKS is 5× worse (173 ms)

What we proved

  1. control_location=back-end is a lie in OpenStack 2024.2 — libvirt XML shows engine='qemu' for both front-end and back-end. librbd native encryption is NOT invoked. Performance is identical.
  2. QEMU multiqueue (queues=4) + iothreads don't help — block/crypto.c has a shared lock per disk image. More queues = more entry points into the same serial bottleneck.
  3. The ~310 MB/s ceiling is one host core saturating at ~125% CPU in the qemu process. Plain uses 182% (more I/O, more librbd threads). dm-crypt uses 482% (4 kernel workers maxed).
  4. Guest dm-crypt delivers 81-93% of plain throughput on large blocks, vs QEMU LUKS at 18-20%. The delta is 4.4×.
  5. On 4k I/O, all four methods are within ±10% — pick whichever has the best key management for your compliance needs.

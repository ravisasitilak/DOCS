LUKS + OpenBao — Operational Flows

Detailed step-by-step flows for encryption, decryption, hot-attach, renewal, migration, and recovery

Section 1: Boot flow (every boot)

When a VM boots (first time or reboot), the luks-agent-boot.service scans all attached block devices and determines whether each volume needs encryption or unlocking. Two separate services provide data at boot: IMDS (token only) and Metadata Service (vendor data + volume flags).

  1. Nova vendor data: Nova serves vendor_data2.json via 169.254.169.254 containing a fetcher script + agent.conf (OpenBao address, custom metadata server address). Kept small — no actual scripts, no secrets, no volume flags.
  2. First boot only: VM boots → cloud-init fetches vendor_data2.json → runs the fetcher script, which downloads luks-agent.sh, systemd units, and udev rules → installs and enables them as systemd services
  3. Subsequent boots: Scripts already installed as systemd services. Vendor data does NOT need to be re-fetched. luks-agent-boot.service starts directly via systemd.
  4. IMDS — token minting: luks-agent calls IMDS POST /v1/token. IMDS verifies VM by source IP, signs JWT (60-second TTL) → calls OpenBao POST /auth/jwt/login → receives BAO token with 1-hour TTL. Writes to /run/bao-token.
  5. luks-agent-boot.service scans all block devices using lsblk
  6. For each device (e.g., /dev/vdX): get Cinder UUID via lsblk -ndo SERIAL /dev/vdX
  7. Query custom metadata server: GET /v1/volumes/{uuid} → returns {encrypt: true/false}
  8. If encrypt=false → skip this device
  9. If encrypt=true and no LUKS header found → Path A (first-time encryption)
  10. If encrypt=true and LUKS header present → Path B (unlock existing)

Section 2: Path A — First-time encryption

When a volume is attached to a VM for the first time, the agent generates a random passphrase, encrypts it with OpenBao Transit, stores the ciphertext in OpenBao KV, then formats the volume with LUKS and creates a filesystem.

  1. Generate 32-byte random passphrase from /dev/urandom and keep in memory only (never written to disk)
  2. Call OpenBao POST /transit/encrypt/luks-master with plaintext passphrase → get ciphertext back
  3. Call OpenBao PUT /luks/data/volumes/{cinder-uuid} → store ciphertext in KV
  4. Call cryptsetup luksFormat --type luks2 --key-file - /dev/vdX → feed passphrase from stdin, format volume with LUKS2 header
  5. Call cryptsetup open --key-file - /dev/vdX crypt-{uuid} → unlock device, create /dev/mapper/crypt-{uuid}
  6. Create filesystem: mkfs.ext4 /dev/mapper/crypt-{uuid}
  7. Mount the encrypted volume: mount /dev/mapper/crypt-{uuid} /mnt/{uuid}
  8. Create LUKS header backup: cryptsetup luksHeaderBackup → store backup (encrypted) in OpenBao PUT /luks/data/headers/{cinder-uuid}
  9. Wipe passphrase from memory (overwrite with zeros)
  10. Add entry to /etc/fstab for automatic mounting: /dev/mapper/crypt-{uuid} /mnt/{uuid} ext4 defaults 0 0

OpenBao API calls: 3 calls total (Transit encrypt + KV write ciphertext + KV write header backup)

Section 3: Path B — Unlock existing volume

When a volume with an existing LUKS header is attached (or VM reboots), the agent retrieves the encrypted passphrase from OpenBao, decrypts it, and unlocks the volume.

  1. Call OpenBao GET /luks/data/volumes/{cinder-uuid} → retrieve ciphertext
  2. Call OpenBao POST /transit/decrypt/luks-master with ciphertext → get plaintext passphrase back
  3. Call cryptsetup open --key-file - /dev/vdX crypt-{uuid} → unlock with passphrase from stdin
  4. Mount the volume: mount /dev/mapper/crypt-{uuid} /mnt/{uuid} (uses existing entry in /etc/fstab from Path A)
  5. Wipe passphrase from memory

OpenBao API calls: 2 calls total (KV read + Transit decrypt)

Section 4: Hot-attach flow

When a volume is attached to a running VM (without reboot), udev detects the new block device and triggers the luks-agent to handle it immediately.

  1. udev detects new block device (e.g., /dev/vdX) → fires rule matching KERNEL=="vd*" → triggers luks-agent@vdX.service
  2. Agent gets Cinder UUID via lsblk -ndo SERIAL /dev/vdX
  3. Agent calls custom metadata server GET /v1/volumes/{uuid} → returns encryption flag (same query as boot flow)
  4. If encrypt=false → skip
  5. If encrypt=true → check if LUKS header exists on device via cryptsetup isLuks /dev/vdX
  6. If no LUKS header → Path A (first-time encryption)
  7. If LUKS header present → Path B (unlock existing)

Note: No reboot needed. Volume is encrypted/unlocked and mounted while the VM is still running.

Section 5: Token renewal

OpenBao tokens have a 1-hour TTL. To avoid expiration during long-running VMs, a systemd timer renews the token every 45 minutes, providing a 15-minute buffer before expiration.

  1. systemd timer fires every 45 minutes (e.g., at 00:00, 00:45, 01:30, ...)
  2. Timer triggers luks-token-renew.service
  3. luks-agent.sh calls IMDS POST /v1/renew
  4. IMDS verifies the request comes from a known VM (source IP + instance UUID in request body)
  5. IMDS creates a new 60-second JWT, calls OpenBao POST /auth/jwt/login, receives a fresh 1-hour BAO token
  6. IMDS returns the new token
  7. Agent writes new token to /run/bao-token (tmpfs, lost on power off)
  8. All future OpenBao calls use the new token
  9. If renewal fails → agent logs error and retries with exponential backoff (e.g., 10s, 20s, 40s)

TTL strategy: Token TTL is 1 hour, renewal fires every 45 minutes. This provides a 15-minute safety buffer. If the IMDS or OpenBao is temporarily unavailable, the token doesn't immediately expire.

Service note: Token renewal calls IMDS only (token minting). It does NOT call the custom metadata server — volume checks happen separately per-volume when the agent runs.

Section 6: Volume migration

A volume can be detached from one VM and attached to another. Because the passphrase is keyed by volume UUID (not VM identity), any authorized VM can decrypt it.

  1. VM-A (source): umount /mnt/{uuid} → unmount the volume
  2. VM-A (source): cryptsetup close crypt-{uuid} → lock the LUKS device
  3. OpenStack admin: openstack server remove volume vm-a volume-uuid → detach from VM-A
  4. OpenStack admin: openstack server add volume vm-b volume-uuid → attach to VM-B
  5. VM-B (destination): udev detects the new device → runs hot-attach flow
  6. VM-B (destination): calls cryptsetup isLuks → detects LUKS header (same UUID)
  7. VM-B (destination): Path B flow: calls OpenBao GET /luks/data/volumes/{uuid} (same UUID) → gets ciphertext
  8. VM-B (destination): calls OpenBao POST /transit/decrypt/luks-master → gets passphrase
  9. VM-B (destination): cryptsetup open + mount → volume is now accessible on VM-B

Why this works: The passphrase is stored in OpenBao KV by volume UUID, not by VM identity. As long as VM-B's OpenBao policy allows reading from /luks/data/volumes/*, it can decrypt any volume it has access to. The LUKS header itself acts as the "proof" that this volume uses LUKS encryption.

Section 7: Disaster recovery

When things go wrong, here's how to recover:

LUKS header corruption

  1. The LUKS header backup was stored in OpenBao during initial encryption: GET /luks/data/headers/{uuid}
  2. Retrieve the backup from OpenBao (it's plaintext — Transit encryption is optional for backups)
  3. Restore header: cryptsetup luksHeaderRestore --header-backup-file /path/to/backup /dev/vdX
  4. Volume is now recoverable

OpenBao unavailable

  1. Volumes stay locked until OpenBao returns
  2. No data loss — the passphrase is safely stored in OpenBao KV, waiting for OpenBao to come back online
  3. Once OpenBao is reachable, token renewal succeeds and volumes can be unlocked normally

Lost BAO token

  1. If /run/bao-token is deleted or corrupted, the VM can no longer make OpenBao API calls
  2. Wait for the next token renewal (every 45 minutes)
  3. Or trigger renewal manually: systemctl start luks-token-renew.service
  4. IMDS issues a new token, and the VM is back online

Volume UUID lookup failure

  1. If lsblk -ndo SERIAL fails or returns empty, the agent can't identify the volume
  2. Manual recovery: use the block device name and manually check for LUKS header: cryptsetup luksDump /dev/vdX
  3. Extract UUID from the LUKS header or from Cinder (e.g., openstack volume show volume-uuid)
  4. Manually retrieve the passphrase: curl -H "X-Vault-Token: $BAO_TOKEN" https://bao.internal/v1/luks/data/volumes/{uuid}
  5. Decrypt the ciphertext: curl -X POST -H "X-Vault-Token: $BAO_TOKEN" -d @payload.json https://bao.internal/v1/transit/decrypt/luks-master
  6. Unlock and mount manually

Section 8: Error handling in luks-agent

The agent implements defensive error handling to prevent partial encryption or orphaned volumes:

BAO token expired

  • Agent checks token expiration time (decoded from JWT)
  • If token is expired: log error to systemd journal, do NOT attempt any OpenBao API calls
  • Agent waits for the next scheduled token renewal (systemd timer fires every 45 min)
  • Once renewed, retries the pending encryption/decryption operation

Transit API unreachable

  • Agent attempts to call POST /transit/encrypt or POST /transit/decrypt
  • If connection times out or returns 5xx: log error, retry with exponential backoff (base 5 seconds, capped at 60 seconds)
  • Max retries: 5 attempts over ~5 minutes total
  • If all retries fail: log device UUID and error details to journal, stop processing (prevents volume from being left half-encrypted)

KV write fails (ciphertext storage)

  • During Path A: agent encrypts passphrase, then tries to store ciphertext in KV via PUT /luks/data/volumes/{uuid}
  • If KV write fails: agent logs error and does NOT proceed to cryptsetup luksFormat
  • Why: if we format the volume but fail to store the passphrase, the volume becomes unrecoverable (orphaned)
  • Agent stops and waits for manual intervention or retry

cryptsetup command fails

  • If cryptsetup luksFormat fails (bad device, I/O error, insufficient space): agent logs device path and UUID to journal
  • Agent does NOT retry automatically — requires manual investigation
  • Error message includes: command output, exit code, device name, volume UUID for recovery

Volume already mounted

  • Agent checks if /mnt/{uuid} already exists and is mounted (via mountpoint command)
  • If already mounted: agent logs "volume already mounted, skipping" and exits cleanly (idempotent)
  • This handles the case where the service is triggered twice at boot or during hot-attach

IMDS temporarily unavailable (token renewal)

  • During token renewal, if IMDS is unreachable: agent retries with backoff
  • Token remains cached in /run/bao-token until it expires (1-hour TTL)
  • Volumes stay mounted and accessible while renewal is being retried
  • Once IMDS returns, next renewal succeeds immediately
  • Note: IMDS is only used for token minting/renewal. Volume flags come from the custom metadata server (queried per-volume at boot and hot-attach)

Journal logging

  • All operations logged to systemd journal via logger or direct syslog
  • View logs: journalctl -u luks-agent-boot.service or journalctl -u luks-agent@vdX.service
  • Logs include: timestamp, severity (INFO/WARN/ERROR), operation (encrypt/decrypt/mount), device, UUID, result
  • For support: SSH into VM and pull journal for troubleshooting

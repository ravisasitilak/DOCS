Analysis & Comparison

Cross-hardware note: Cinder LUKS and guest dm-crypt were benchmarked on Intel (4-vCPU VM, single Ceph OSD). RBD Native LUKS2 on AMD EPYC (bare-metal, rbd-nbd). Nova Ephemeral on Intel i9-14900 (host LVM). Numbers are directional across hardware — the architectural differences (single-thread vs multi-thread, hypervisor vs kernel, QEMU vs librbd) are what matter.

Why the massive spread: 0% to 81% overhead across options

All 6 options use the same AES-XTS-256 cipher with AES-NI hardware acceleration. The cipher math is ~6 µs per 4 KiB sector on any of these CPUs. The overhead differences come entirely from where in the stack the cipher runs and how many threads do the work.

OptionThreadingWhy it's fast or slow
Cinder LUKS1 QEMU iothreadAll crypto serialized through block/crypto.c on a single coroutine. Can't parallelize. Multiqueue + iothreads don't help (shared mutex per disk image). Caps at ~310 MB/s.
RBD Native LUKS222 rbd-nbd workerslibrbd's CryptoObjectDispatch runs AES in the daemon's worker thread pool. Multi-threaded, but also benefits from staying in userspace (no kernel block layer hops).
Guest dm-cryptN kcryptd workers (N=vCPUs)Kernel dm-crypt creates one worker per CPU. With 4 vCPUs, 4 parallel crypto pipelines. Crypto happens BEFORE data leaves the guest — QEMU sees ciphertext, passes it straight through.
Nova EphemeralN kcryptd workers (N=host CPUs)Same dm-crypt as guest, but on the HOST kernel. More CPUs available. Bottleneck shifts to NVMe sequential read speed with default config; NQ+4096 optimization removes queue overhead.
Ceph OSD dm-cryptN host kernel workersRuns below BlueStore's cache. Cache absorbs read/write patterns, issues large sequential I/O to dm-crypt. Overhead hidden by caching — workloads never see it.

The single-thread vs multi-thread gap — measured

CINDER LUKS (1 thread, 310 MB/s):
  iothread: [encrypt]→[submit]→[WAIT]→[encrypt]→[submit]→[WAIT]→...
  CPU: 125% (1 core maxed, 3 idle)
GUEST DM-CRYPT (4 threads, 1,391 MB/s):
  kcryptd/0: [encrypt #1]──[encrypt #5]──...
  kcryptd/1: [encrypt #2]──[encrypt #6]──...
  kcryptd/2: [encrypt #3]──[encrypt #7]──...
  kcryptd/3: [encrypt #4]──[encrypt #8]──...
  CPU: 482% (all 4 cores doing useful work)
Same cipher. Same AES-NI. Same Ceph. 4× parallelism = 4.4× throughput.

Why control_location=back-end didn't help

We expected Cinder's control_location=back-end to use librbd's multi-threaded crypto (like RBD Native). It doesn't. The libvirt XML for both front-end and back-end shows engine='qemu' — same single-threaded QEMU code path. This is a missing feature in Nova's libvirt driver in OpenStack 2024.2, not a misconfiguration.

Why multiqueue + iothreads didn't help

We tested QEMU queues=4 + dedicated iothread per disk. Result: still 310 MB/s. block/crypto.c uses a shared crypto context (mutex) per disk image. More queues = more entry points into the same lock.

Nova Ephemeral Disk Encryption — Architecture & Optimization

What: Encrypts the VM's root disk (/dev/vda) and ephemeral scratch disks on the compute host using host-side dm-crypt. The VM has no idea encryption is happening — QEMU sees the dm-crypt mapper device as a regular block device.

How it works

Without encryption:
  VM (QEMU) ──► /dev/nova-volumes/<uuid>_disk ──► Physical Disk
With encryption:
  VM (QEMU) ──► /dev/mapper/<uuid>_disk-dmcrypt ──► /dev/nova-volumes/<uuid>_disk ──► Physical Disk
                        │
                   dm-crypt layer
                 (AES encrypt/decrypt
                  in kernel RAM)

Key Management with OpenBao (Vault fork)

Step 1:  User runs "openstack server create"
Step 2:  nova-api sees ephemeral_storage_encryption = True
Step 3:  nova-api calls OpenBao: POST /v1/secret/data/<new-uuid>
Step 4:  OpenBao generates random 256-bit key, stores in KV store
Step 5:  nova-api saves ONLY the key UUID in Nova's database
Step 6:  nova-api sends build request to nova-compute (includes key UUID)
Step 7:  nova-compute calls OpenBao: GET /v1/secret/data/<key-uuid>
Step 8:  OpenBao returns actual key bytes
Step 9:  nova-compute derives dm-crypt key: SHA-256(hexlify(OpenBao_key))
Step 10: nova-compute passes key to cryptsetup via stdin
         Key lives in kernel RAM. ZERO more calls to OpenBao.
VM Deletion:
Step 11: cryptsetup remove (mapper removed, key wiped from RAM)
Step 12: lvremove (LV deleted from disk)
Step 13: DELETE key from OpenBao (purged permanently)

How to deploy (cryptsetup wrapper)

Nova hardcodes cryptsetup create without optimization flags. The wrapper intercepts the call and injects them:

#!/bin/bash
# /opt/cryptsetup-wrapper/cryptsetup
if [ "$1" = "create" ]; then
    shift
    NAME="$1"; shift
    DEVICE="$1"; shift
    exec /usr/sbin/cryptsetup.real open --type plain "$DEVICE" "$NAME" \
        --sector-size 4096 \
        --perf-no_read_workqueue \
        --perf-no_write_workqueue \
        "$@"
else
    exec /usr/sbin/cryptsetup.real "$@"
fi

# Deploy into nova_compute container:
docker cp nova_compute:/usr/sbin/cryptsetup /opt/cryptsetup-wrapper/cryptsetup.real
docker cp /opt/cryptsetup-wrapper/cryptsetup nova_compute:/usr/sbin/cryptsetup
# Verify on new VMs:
sudo dmsetup table <uuid>_disk-dmcrypt
# Should show: ... no_read_workqueue no_write_workqueue sector_size:4096
Caveat: The wrapper must be reapplied after any kolla-ansible redeploy. Existing VMs must be recreated to pick up the new flags (set at dm-crypt device creation time, not changeable on running devices).

Key Findings

1. 4k I/O: encryption is essentially free
All 4k workloads show ±5% IOPS delta. The per-block crypto cost (~20-50 µs) is invisible against the ~500 µs Ceph round-trip. For OLTP/database workloads, Cinder LUKS adds negligible overhead.
2. 1m/4m I/O: 75-85% throughput loss
Encrypted throughput flatlines at ~310-350 MB/s while plain scales to 1.5-2.4 GB/s. The QEMU single-iothread serializes all crypto, creating a hard ceiling regardless of iodepth.
3. The ~310 MB/s ceiling is CPU-bound
pidstat shows encrypted 1m/4m workloads use ~125% CPU (one host core maxed). Plain uses 155-185%. Encrypted uses LESS CPU but delivers LESS throughput — most of the system is idle, blocked behind a single-thread chokepoint.
4. iodepth doesn't help encrypted large-block I/O
enc 1m write: depth=8→320 MB/s, depth=16→317 MB/s, depth=32→316 MB/s. Flat. The QEMU iothread can't parallelize crypto. More outstanding I/Os just increase tail latency.
5. Three iterations confirm stability
Encrypted throughput varies <3% across iterations (310±10 MB/s). Plain varies 5-15%. Iteration rotation cancels first-run bias. Results are reproducible.

Anomaly Analysis

The unified mechanism: BlueStore _kv_sync_thread sleep gap

Three cells show encryption running FASTER than plain: 1m_seqwrite qd8 (+14.4%), 4m_seqwrite qd1 (+9.7% refined), 4m_randwrite qd1 (+12.4%). All three are caused by the same single mechanism in Ceph's BlueStore commit pipeline.

BlueStore has a single-threaded RocksDB metadata commit thread (_kv_sync_thread). Its loop:

loop forever:
  1. grab everything in kv_queue          (a batch)
  2. release the throttle                 (let producers send more)
  3. submit_transaction_sync(...)         (slow fsync, ~30 ms)
  4. mark txcs done, wake clients
  5. loop back to step 1
     → if kv_queue is empty: SLEEP        ← THE GAP
       until somebody puts a new txc in

Why encryption fills the gap

PLAIN (loses the race — fio bursts, OSD sleeps in the gaps):
  fio:    ████████                  ████████                  ████████
          burst                     burst                     burst
  osd:    [batch]──fsync──┐         [batch]──fsync──┐
                          │ZZZ sleep│                │ZZZ sleep│
                          └─────────┘                └─────────┘
                           dead air                   dead air
LUKS2 (wins the race — cipher cost paces submissions):
  fio:    █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █
          cipher cost spreads IOs ~1.5 ms apart (1 MiB) or ~6 ms (4 MiB)
  osd:    [batch]──fsync──[batch]──fsync──[batch]──fsync──
                          ↑              ↑              ↑
                          already here   already here   already here
                          no sleep       no sleep       no sleep

Why this matters

The passphrase file sits on the compute host at a known path. To map an encrypted volume:

sudo rbd-nbd map --encryption-format luks2 \
    --encryption-passphrase-file /path/to/luks.pass \
    pool/image

That file must exist on every compute host that might run a VM using this volume. There is no mechanism to:

  • Fetch the key dynamically at attach time (like Cinder + OpenBao does)
  • Scope the key to a specific tenant or project
  • Revoke the key remotely (you must delete the file from every host)
  • Audit who mapped which image with which key
  • Rotate keys without downtime (must unmap, reformat, remap)

Bottom line on KMS

rbd-nbd native LUKS2 delivers the best raw encryption performance (1–3% overhead) but cannot be used in any environment that requires centralized key management, auditing, or multi-tenant key isolation. For those requirements, use guest dm-crypt with OpenBao/Vault/Tang (19% overhead on large blocks) or accept Cinder LUKS's performance penalty (81% on large blocks) in exchange for native OpenBao integration.

Verdict

Performance: Ceph RBD native LUKS2 encryption via rbd-nbd can be enabled with negligible performance impact on AES-NI hardware. The expected throughput cost is approximately 1–3% across all measured workloads. The performance question should not be a deciding factor.
Operations: The absence of KMS integration means this path is suitable only for environments where passphrase-file-on-disk is an acceptable key management model. For production OpenStack clouds requiring OpenBao, Vault, or any centralized KMS, this path cannot be used without custom integration work that does not exist upstream.

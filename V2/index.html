LUKS + OpenBao Volume Encryption

Guest-level LUKS encryption with OpenBao Transit + KV, triggered by Cinder volume type, deployed via vendor data

Admin setup

1. One-time admin setup

Everything the cloud admin configures once. After this, no more admin action needed.

flowchart TD
    A1["1. Create Cinder volume type\nopenstack volume type create secure-volume\nopenstack volume type set secure-volume\n--property openbao_encrypt=true"] --> A2

    A2["2. Configure OpenBao\nEnable Transit engine + create key luks-master\nEnable KV store at luks/\nEnable AppRole auth\nCreate VM access policies"] --> A3

    A3["3. Create vendor_data.json\nContains cloud-config YAML with:\nluks-agent.sh script as base64\nsystemd unit files\nudev rules\nBAO address + CA cert\nNO secrets in this file"] --> A4

    A4["4. Update nova.conf on all API servers\nvendordata_providers = StaticJSON\nvendordata_jsonfile_path = /etc/nova/vendor_data.json\nRestart nova-api"] --> A5

    A5["5. Add IMDS endpoint\nGET /v1/instance/volumes\nQueries Cinder API for volume types\nReturns encrypt true or false per volume"] --> DONE["Done. Infrastructure ready."]

How vendor data works

Nova serves vendor data via metadata service. cloud-init inside the VM reads it automatically and installs luks-agent. Works with any image. Customer does nothing.

Authentication

3. Auth flow — VM gets OpenBao credentials

VM calls custom IMDS once per boot. Gets AppRole credentials, logs into OpenBao, caches the token. No OpenStack credentials inside the VM.

sequenceDiagram
    participant VM as Guest VM
    participant IMDS as Custom IMDS
    participant BAO as OpenBao

    Note over VM: Boot or cached token expired

    VM->>IMDS: GET /v1/openbao/creds
    Note over IMDS: Verify VM identity
(instance UUID / source IP)
    IMDS->>BAO: Generate secret_id for AppRole
    BAO-->>IMDS: secret_id (TTL, single-use)
    IMDS-->>VM: role_id + secret_id

    VM->>BAO: POST /auth/approle/login
    BAO-->>VM: token (24h TTL, renewable)
    Note over VM: Cache token at /run/bao-token
(tmpfs, mode 0600, root only)

    loop Every 8 hours
        VM->>BAO: POST /auth/token/renew-self
        BAO-->>VM: renewed token
    end

    Note over VM: If renewal fails then call IMDS again

Volume encryption decision

4. Per-volume decision via IMDS

Single config point: Cinder volume type. luks-agent asks IMDS, IMDS queries Cinder for the volume type. Only volumes with openbao_encrypt=true get encrypted.

sequenceDiagram
    participant VM as luks-agent
    participant IMDS as Custom IMDS
    participant Cinder as Cinder API

    VM->>VM: lsblk -ndo SERIAL /dev/vdX
Gets Cinder volume UUID

    VM->>IMDS: GET /v1/instance/volumes
    Note over IMDS: IMDS has Cinder admin credentials
VM does NOT access Cinder directly

    IMDS->>Cinder: GET /v3/volumes for this instance
    Cinder-->>IMDS: Volume list with types

    IMDS->>Cinder: GET volume type extra_specs
    Cinder-->>IMDS: openbao_encrypt = true or false

    IMDS-->>VM: vol-abc encrypt true
vol-def encrypt false

    VM->>VM: vol-abc → proceed to encrypt or decrypt
    VM->>VM: vol-def → skip

Encryption flows

5. Master decision flow

Two triggers (boot, hot-attach) feed into one decision tree. Check volume type via IMDS then check if LUKS header exists. Branch into encrypt new or unlock existing.

flowchart TD
    T1["Trigger 1: systemd boot.service\n(every boot)"] --> SCAN["Scan all block devices\nlsblk -dpno NAME,TYPE"]
    T2["Trigger 2: udev rule\n(hot-attach, no reboot)"] --> SINGLE["Single new device /dev/vdX"]

    SCAN --> FOREACH["For each /dev/vd(b..z)"]
    SINGLE --> FOREACH
    FOREACH --> SERIAL["Get Cinder UUID via lsblk SERIAL"]

    SERIAL --> TOKEN{"Cached BAO token\nat /run/bao-token?"}
    TOKEN -->|Yes| USE["Use cached token"]
    TOKEN -->|No| IMDS_AUTH["Call IMDS for AppRole creds\nLogin to BAO and cache token"]

    USE --> IMDS_VOL
    IMDS_AUTH --> IMDS_VOL

    IMDS_VOL["Call IMDS /v1/instance/volumes\nGet encryption flag for this volume"]

    IMDS_VOL --> CHECK_TYPE{"Volume type has\nopenbao_encrypt = true?"}
    CHECK_TYPE -->|No| SKIP["Skip this volume"]

    CHECK_TYPE -->|Yes| CHECK_LUKS{"cryptsetup isLuks?"}
    CHECK_LUKS -->|"No"| PATH_A["Path A: First-time encrypt"]
    CHECK_LUKS -->|"Yes"| PATH_B["Path B: Decrypt and open"]

    PATH_A --> A1["Generate random passphrase\nhead -c 32 /dev/urandom"]
    A1 --> A2["Transit encrypt passphrase"]
    A2 --> A3["Store ciphertext in BAO KV"]
    A3 --> A4["cryptsetup luksFormat"]
    A4 --> A5["cryptsetup open"]
    A5 --> A6["mkfs.ext4 + mount"]
    A6 --> A7["Backup LUKS header to BAO KV"]
    A7 --> DONE_A["Wipe passphrase + update fstab"]

    PATH_B --> B1["Read ciphertext from BAO KV"]
    B1 --> B2["Transit decrypt passphrase"]
    B2 --> B3["cryptsetup open"]
    B3 --> B4["mount"]
    B4 --> DONE_B["Wipe passphrase"]

6. Path A — first-time encryption

New volume with no LUKS header. Generate passphrase, encrypt via Transit, store ciphertext in KV, format and mount the volume.

sequenceDiagram
    participant VM as Guest VM
    participant TR as OpenBao Transit
    participant KV as OpenBao KV Store
    participant VOL as Volume /dev/vdX

    Note over VM: encrypt=true and isLuks=No

    Note over VM: Generate passphrase
head -c 32 /dev/urandom
(memory only, never on disk)

    VM->>TR: POST transit/encrypt/luks-master
    Note over TR: Encrypts with internal key
Does NOT store anything
    TR-->>VM: ciphertext vault:v1:8SD...

    VM->>KV: PUT luks/data/volumes/VOL_ID
Store ciphertext
    Note over KV: Stored permanently by volume UUID

    VM->>VOL: cryptsetup luksFormat --type luks2 --key-file -
    Note over VOL: LUKS header created

    VM->>VOL: cryptsetup open
    VM->>VOL: mkfs.ext4 + mount /mnt/UUID
    VM->>KV: Backup LUKS header

    Note over VM: unset PASS
Passphrase exists NOWHERE now

7. Path B — unlock existing volume

Volume already has LUKS header. Read ciphertext from KV, decrypt via Transit, open and mount.

sequenceDiagram
    participant VM as Guest VM
    participant KV as OpenBao KV Store
    participant TR as OpenBao Transit
    participant VOL as Volume /dev/vdX

    Note over VM: encrypt=true and isLuks=Yes

    VM->>KV: GET luks/data/volumes/VOL_ID
    KV-->>VM: ciphertext vault:v1:8SD...

    VM->>TR: POST transit/decrypt/luks-master
    TR-->>VM: plaintext passphrase

    VM->>VOL: cryptsetup open --key-file -
    VM->>VOL: mount /mnt/UUID

    Note over VM: unset PASS
Volume stays mounted until reboot

Hot-attach and migration

8. Hot-attach flow

Volume attached to a running VM. udev detects it, triggers luks-agent. No reboot needed. Same logic as boot scan.

sequenceDiagram
    participant Cust as Customer
    participant Nova as Nova
    participant VM as Running VM
    participant IMDS as IMDS
    participant BAO as OpenBao

    Cust->>Nova: openstack server add volume my-vm my-secure-vol
    Nova->>VM: Attach volume
    Note over VM: /dev/vdc appears

    VM->>VM: udev rule fires
systemctl start luks-agent@vdc
    VM->>VM: Get Cinder UUID via lsblk SERIAL
    VM->>IMDS: GET /v1/instance/volumes
    IMDS-->>VM: vol-uuid encrypt true

    VM->>VM: isLuks? No
    VM->>BAO: Transit encrypt + KV store
    VM->>VM: luksFormat + open + mkfs + mount

    Note over VM: /mnt/vol-uuid ready
No reboot needed

9. Volume migration

Detach from VM-A, attach to VM-B. Passphrase is stored by volume UUID in OpenBao. Any authorized VM can decrypt it.

sequenceDiagram
    participant A as VM-A
    participant OS as Nova / Cinder
    participant BAO as OpenBao
    participant B as VM-B

    A->>A: umount + cryptsetup close
    A->>OS: Detach volume
    OS->>B: Attach volume to VM-B
    Note over B: /dev/vdX appears

    B->>B: lsblk SERIAL then cryptsetup isLuks = Yes

    B->>BAO: GET luks/data/volumes/UUID
    BAO-->>B: ciphertext
    B->>BAO: POST transit/decrypt
    BAO-->>B: passphrase

    B->>B: cryptsetup open + mount
    Note over B: Same volume, different VM
Same passphrase via OpenBao

OpenBao internals

10. Inside OpenBao

One server, multiple engines at different URL paths. Transit and KV are independent. Everything barrier-encrypted on disk. Token + policy required for access.

flowchart TD
    subgraph BAO["OpenBao — https://openbao:8200"]
        subgraph AUTH["Auth"]
            AR["AppRole\n/auth/approle/login\nReturns token"]
        end

        subgraph SECRETS["Secret engines"]
            TR["Transit\n/transit/encrypt + /decrypt\nEncrypt/decrypt passphrase\nKey in MEMORY only\nStores NOTHING"]
            KV["KV Store\n/luks/data/volumes/uuid\nStores ciphertext\nStores LUKS header backups"]
        end

        subgraph SYS["System"]
            POLICY["Policies\nVM: KV read/write + Transit en/decrypt\nAdmin: KV read only\nSeparate permissions"]
            BARRIER["Barrier encryption\nALL data encrypted on disk\nProtected by unseal keys"]
        end
    end

    VM["Guest VM"] -->|"Same token\ndifferent paths"| AUTH
    VM --> SECRETS
    TR -.->|"Independent"| KV
    KV --> BARRIER
